# 股票深度研究（9 段框架）

## 概述

对 A 股上市公司进行结构化深度研究，输出 9 段投研素材包。框架本身与数据源无关，优先使用 iFinD MCP 获取结构化财务与量化数据，用 AlphaClaw 获取定性分析（研报/纪要/公告）。

**执行模式**：两步确认 — 预研后先输出轻量框架请你确认，再深度写报告。确认后根据体量自动选择单 agent 或 3 簇并行写作。

## 执行铁律

| # | 规则 | 说明 |
|---|------|------|
| 1 | **并行** | Step 2 的 13 条数据命令、Step 4b 的 3 个段簇 sub-agent，必须在同一条消息中并行发起 |
| 2 | **确认** | Step 3 预研框架 + 用户确认不允许跳过（用户说"直接出报告"例外） |
| 3 | **溯源** | 每条结论标注三要素：证据等级（F1/F2/M1/C1/H1）+ 来源工具/文件 + 数据时间 |
| 4 | **分工** | 编排模式下主 agent 定首席论点 + 段级论点，sub-agent 写段正文，禁止主 agent 越权写段 1-7 |
| 5 | **纯文本** | 全文只用 markdown 原生格式（文字/表格/列表），禁止生成任何图片/图表/可视化；主体内容 bullet point 驱动 |

## 金字塔分工模型

编排模式下，金字塔按层级在主 agent 和 sub-agent 间分工：

```
首席投资论点（主 agent，Step 3）
  ├── MECE + 逻辑链验证（主 agent，Step 3.5）← 框架层面自检，不通过不进入 Step 4
  ├── 段1论点 → 段1正文（Sub-agent A，Step 4b）
  ├── 段2论点 → 段2正文（Sub-agent A）
  ├── 段3论点 → 段3正文（Sub-agent A）
  ├── 段4论点 → 段4正文（Sub-agent A）  ← 簇A：基本面链
  ├── 段5论点 → 段5正文（Sub-agent B，Step 4b）
  ├── 段6论点 → 段6正文（Sub-agent B）    ← 簇B：竞争力市场链
  ├── 段7论点 → 段7正文（Sub-agent C，Step 4b） ← 簇C：对标链
  ├── 段8 跟踪指标（主 agent，Step 4c）
  ├── 段9 待补资料（主 agent，Step 4c）
  ├── 引言 + 跨段衔接（主 agent，Step 4c）
  └── Acceptance Self-check（主 agent 内部自检，Step 4c，不输出到报告）
```

| 层级 | 负责人 | 产物 | 输入 |
|------|--------|------|------|
| 首席论点 + 段级论点 | 主 agent（Step 3） | 预研框架（含段级论点 7 句） | Step 2 全部数据 + 预热结论 |
| 段 1-7 正文 | 3 个 sub-agent 并行（Step 4b） | 3 个段簇文件 | 段级论点 + 分配的数据 + analysis-template 对应分区 |
| 段 8-9 + 引言 + 衔接 + 自检 | 主 agent（Step 4c） | 最终完整报告 | 全部段簇输出 |

**段簇划分逻辑**：

| 簇 | 段 | 依赖链 | 共享数据源 | Sub-agent |
|----|-----|--------|-----------|-----------|
| A 基本面链 | 段1→2→3→4 | 段内强顺序依赖（定位→拆分→驱动→利润） | iFinD 财务 + 行业宏观 | Sub-agent A（顺序写） |
| B 竞争力市场链 | 段5→6 | 段5 识别风险 → 段6 检验定价 | AlphaClaw 定性 + iFinD 股权/风险 | Sub-agent B |
| C 对标链 | 段7 | 弱依赖前段（只需要可比公司财务数据） | iFinD 可比公司批量财务 | Sub-agent C |

段 8-9 归主 agent：两者都是从全部段输出中提炼的（"该跟踪什么""还缺什么"），每个 300-500 词。主 agent 在 4c 读完三个簇后自然知道内容。

## 时间精度规则

> 在构造所有数据查询（iFinD / AlphaClaw）时，必须根据用户的时间表述选择正确的精度。

| 用户表述 | 查询精度 | iFinD time_start/end | AlphaClaw date_range |
|---------|---------|---------------------|---------------------|
| "今天/盘后/盘中/这两天" | 日级别 | 当日日期 | `past_day` |
| "本周/这周/过去一周" | 周级别 | 本周一 ~ 当日 | `past_week` |
| "本月/这个月/近一个月" | 月级别 | 本月1日 ~ 当日 | `past_month` |
| "最近/近期/最新"（无明确时间单位） | 季度级别 | 近3个月 | `past_quarter`（默认） |
| "本季度/近三个月" | 季度级别 | 当前季度起始 | `past_quarter` |
| "近半年" | 半年级别 | 近6个月 | `past_half_year` |
| "近一年/去年" | 年级别 | 对应年份 | `past_year` |

**原则**：用户说了具体时间单位 → 必须用对应精度，禁止扩大范围。

## 9 段结构

1. **公司定位**：产业链位置 / 产品结构 / 差异化 vs 同业
2. **业务拆分**：各业务线收入/占比/YoY/毛利率
3. **收入驱动**：量/价/结构/客户四维拆解
4. **利润驱动**：盈利能力 / 弹性分析 / 一致预期
5. **核心竞争力与主要风险**：逐项 vs 可比公司 + 量化支撑
6. **市场焦点**：催化逻辑 / 核心担忧 / 预期差 / **情绪位置**（卖方分歧度/持仓拥挤度/媒体热度）/ **不对称性赔率**（乐观上行空间 vs 悲观下行空间）
7. **可比与差异**：关键指标横向对比（≥3家）/ **安全边际检查**（调整后内在价值锚定 vs 当前股价、历史安全边际分位、当前安全边际充足度判断）
8. **跟踪指标**：下次财报重点 / 三个月关键变量
9. **仍需补的资料**：必需 / 建议 / 不确定但影响判断

## 证据等级

| 标签 | 含义 | 示例 |
|------|------|------|
| F1 | 公开可验证事实 | 交易所披露 |
| F2 | 财报/金融数据库/公告 | iFinD 数据、年报 |
| M1 | 市场一致预期 + 管理层公开发言 | 卖方预测、业绩会发言 |
| C1 | 合理推演 | 需附推演链路 |
| H1 | 待核验假设 | 必须标"未经核验" |

**约束**：没有证据等级不能写。关键结论必须有 F1/F2 支撑。管理层公开发言 = M1，不是 F2。

## 输出格式禁止

- **禁止生成图片/图表/可视化**：全文仅用 markdown 原生格式（文字、表格、列表）
- **表格替代图表**：需要对比数据时用 markdown 对比表格，不生成柱状图/折线图/饼图
- **列表替代流程图**：需要展示流程/层级时用嵌套列表，不生成流程图/思维导图
- **禁止 ASCII art**：不使用字符画、分隔线装饰等非内容元素
- **bullet point 优先**：主体内容用 bullet point 短句，每个信息点 2-3 行内讲清；仅引言和结语允许连续段落体
- **每点独立可扫**：一个 bullet 只承载一个信息维度；需展开时用子级 bullet 缩进

## 数据源分工原则

> **推荐调用方式**：所有 iFinD MCP 接口优先使用 `mcporter call`（`tool` 参数 = `"服务器.工具"` 点号连接格式），详见 `references/data-commands.md`。bash + call.py 仅在 mcporter 不可用时作为备选。

| 数据类型 | 工具 | 调用方式 |
|---------|------|---------|
| 结构化财务、估值、风险、股权、事件、行情 | `hexin-ifind-ds-stock-mcp.*` | `mcporter call tool="hexin-ifind-ds-stock-mcp.<方法>"` |
| 行业宏观（产销量/价格） | `hexin-ifind-ds-edb-mcp.*` | `mcporter call tool="hexin-ifind-ds-edb-mcp.<方法>"` |

## 执行步骤

### 第一步：研究预热（Broad → Narrow）

**目的**：在正式调数据之前，快速建立对这家公司的整体认知，确定重点关注维度。

并行执行以下操作：
1. 调用 `search_news` 查询该公司近 3 个月的新闻，了解当前市场焦点
2. 根据新闻结果，识别该公司当前最核心的争议点（催化逻辑 vs 市场担忧）

**预热结论格式**：
```
【研究预热结论】
- 当前市场最关注：[核心议题]（来源：[新闻日期]）
- 核心担忧：[市场担心的点]
- 建议重点深挖：[1-2个需要重点验证的维度]
```

预热完成后进入第二步。如果预热发现公司近期无重大事件或新闻稀少，继续进入第二步（不阻塞）。

### 第二步：提取证券信息 + 并行取数

从用户消息中提取公司名称和证券代码。如果用户只给了公司名没给代码，用 `iFinD search_stocks` 查询。

> **执行前**：读取 `references/data-commands.md` 获取 iFinD bash 命令模板和 AlphaClaw 参数表。
> 数据源不确定时 → 读取 `references/quick-ref.md` 的"工具快速参考"表。

**同时在同一条消息中并行执行所有以下命令**。将 `<公司名>`、`<今日日期>`、`<近3个月日期>` 替换为实际值后，在 `D:\Alphaclaw\` 目录下执行 `references/data-commands.md` 中的全部命令模板：iFinD 股票数据 7 条 + iFinD 新闻 2 条 + 趋势新闻 1 条 + AlphaClaw 定性分析 4 条，共 13 条并行发出。

> **数据优先级说明**：P0 = 缺失则报告无法完成；P1 = 重要增强；P2 = 有则更好。

### 第三步：生成预研框架 + 段级论点，发给你确认

> **执行前**：读取 `references/framework-confirm.md` 获取预研框架模板。

按 `references/framework-confirm.md` 中的模板生成轻量预研框架，内容包括：

- 研究预热结论（第一步的输出）
- 数据获取概况（成功/缺失）
- **投资论点金字塔**：首席投资论点（塔尖）+ 3-4 个支撑性子论点 + 风险反向论点 + 一句话投资命题
- **段级论点（新增）**：对段 1-7 各写一句塔尖断言。这句断言将作为 Step 4b sub-agent 写作时的"段塔尖"，sub-agent 围绕它展开正文
- 建议的可比公司名单（≥3家，附选择理由）
- 当前市场核心争议（催化逻辑 vs 市场担忧）
- **询问你**：
  1. 研究重点是否需要调整？
  2. 有没有要补充的数据或要换的可比公司？
  3. 有没有特别关注的问题希望在报告里重点分析？

**段级论点写作规范**：
- 每段一句断言句，不用主题描述句。❌ "公司收入驱动分析" ✅ "收入增长由 NOR Flash 量价齐升驱动，MCU 补库存周期提供第二曲线"
- 7 句之间逻辑自洽：段 2 的论点要从段 1 延伸，段 3 从段 2 延伸，以此类推
- 如果某段数据不足、无法形成断言，标注 `[待数据补充]`

**等你回复确认后，再进入第四步。**

### 写作前综合检查（确认后、动笔前）

> 确认后、动笔前，先自问以下 7 条。任一条为"否"则先补充检索，不急于动笔。

| # | 检查项 | 通过标准 |
|---|-------|---------|
| 1 | 多角度覆盖 | 已从 ≥3 个角度检索（财务/研报/新闻/行业/可比）？ |
| 2 | 核心数据就位 | P0 级数据覆盖率 ≥ 80%？段1-段7 每个关键结论有 F2 级数据锚点？ |
| 3 | 正反观点完整 | 同时检索了正面催化逻辑和负面担忧/质疑观点？ |
| 4 | 核心争议可回答 | 预研阶段发现的核心争议点有数据可以回答？ |
| 5 | 信息时效合格 | 财务数据来自最新财报期？研报/纪要来自近3个月？ |
| 6 | 格式合规 | 全文无图片/图表/可视化？仅用 markdown 原生格式？主体内容 bullet point 驱动？ |
| 7 | Fact Hub 交叉引用 | 已检查 Fact Hub 中相关主题的 opinions.md，识别可应用的跨赛道分析框架？若无 Fact Hub 知识库或对应主题为空 → 自动跳过 |

→ **任一条未通过**：先补充检索，不急于动笔。全部通过后进入 Step 3.5。

### 第三步补充：MECE + 逻辑链验证（Step 3.5）

> **目的**：动笔前对金字塔框架做最后一轮结构自检。发现问题改框架，发现不了再动笔——这是避免"报告写完才发现逻辑链断了"的根本手段。

#### 三维验证框架

**维度一：MECE 检查**（Mutually Exclusive, Collectively Exhaustive）

对首席论点 + 7 个段级断言逐一核查：

| 检查项 | 通过标准 | 未通过处理 |
|-------|---------|----------|
| 首席论点与段1-7 是否覆盖了投资判断的全部核心维度？ | 至少覆盖：基本面（段1-4）+ 竞争力（段5）+ 市场定价（段6）+ 相对估值（段7） | 缺哪个维度 → 补充对应段级断言 |
| 7 个段级论点之间是否有重叠（同一事实被两个段同时论证）？ | 每段论证重心不重合，段间衔接靠"承接前段结论"而非"重复前段数据" | 有重叠 → 明确各段边界，重新划定论证重心 |
| 风险论点是否独立于正向论点（不是正向论点的反面，而是独立的额外风险）？ | 风险是独立威胁，不是"增长不及预期"这种显而易见的反向陈述 | 风险与正向论点重叠 → 拆解为更具体的独立风险 |

**维度二：逻辑链完备性检查**

| 检查项 | 通过标准 | 未通过处理 |
|-------|---------|----------|
| 段1（定位）→ 段2（业务拆分）：定位中识别的业务线，段2是否全部有财务数据支撑？ | 段1列出的每条业务线，段2都有对应收入/毛利率数据 | 缺业务线数据 → 段2论点降级为"[待数据补充]"，段9 列获取路径 |
| 段2（业务拆分）→ 段3（收入驱动）：段2识别的增长业务，段3是否追溯到了量/价/结构/客户的具体驱动因子？ | 增长业务的 YoY 能找到量或价的驱动因子，无断裂 | 驱动因子缺失 → 段3论点标注"[需补]"，回退至数据补充 |
| 段3（收入驱动）→ 段4（利润驱动）：收入增速是否在段4有对应的利润率验证（不是"增收不增利"的陷阱）？ | 段4的毛利率/净利率趋势与段3的驱动因子逻辑一致 | 增收不增利 → 在首席论点中补充利润率约束条件 |
| 段4（利润驱动）→ 段5（竞争力）：高 ROE/利润率是护城河的结果（段5验证），还是昙花一现？ | 段5的 VRIO 分析回应了段4 ROE 的可持续性问题 | 逻辑断裂 → 在段5论点中补充护城河验证方向 |
| 段5（竞争力）→ 段6（市场焦点）：段5识别的核心风险，段6是否检验了市场是否已定价？ | 段6的催化/担忧逻辑与段5的核心风险直接对应 | 段6未回应段5风险 → 段6论点补充"市场定价检验"方向 |
| 段6（市场焦点）→ 段7（可比差异）：估值溢价/折价的原因是否在段7有数据锚点？ | 段7的估值归因回应了段6的市场定价逻辑 | 估值归因空洞 → 段7论点锚定具体指标差异 |

**维度三：首席论点与段级论点的对齐检查**

| 检查项 | 通过标准 | 未通过处理 |
|-------|---------|----------|
| 首席论点是"看多/看空/待观察"是否由段1-7的论点共同支撑？ | 每个子论点对应至少一个段级断言 | 有论点无段落支撑 → 删除该子论点或补充对应段 |
| 风险反向论点是否在段5或段6中有具体触发条件和量化锚点？ | 风险有触发条件 + 数据锚点，不是泛泛而谈 | 风险无锚点 → 在段5或段6补充风险量化验证方向 |

#### 执行方式

**第一轮（主 agent 自检，必做）**：

主 agent 读取 Step 3 输出的预研框架（首席论点 + 7 句段级断言），对照上述三维度逐一打勾，输出验证报告：

```
【Step 3.5 验证报告】

■ MECE 检查：
  ✅/❌ 首席论点覆盖全部核心维度 → [结论]
  ✅/❌ 7句无重叠 → [结论]
  ✅/❌ 风险论点独立于正向论点 → [结论]

■ 逻辑链检查：
  ✅/❌ 段1→段2：业务线全覆盖 → [结论]
  ✅/❌ 段2→段3：增长驱动因子追溯 → [结论]
  ✅/❌ 段3→段4：利润率验证一致 → [结论]
  ✅/❌ 段4→段5：ROE 护城河验证方向 → [结论]
  ✅/❌ 段5→段6：市场定价检验方向 → [结论]
  ✅/❌ 段6→段7：估值归因锚点 → [结论]

■ 对齐检查：
  ✅/❌ 首席论点有段级支撑 → [结论]
  ✅/❌ 风险论点有量化锚点 → [结论]

【综合判断】：✅ 通过 / ⚠️ 存在 N 处问题需修正

[如有问题，逐条列出修正方案]
```

**第二轮（可选，依赖判断）**：

若第一轮发现逻辑链断裂（❌ ≥ 3 处），在修正后向用户发一封简短确认：
> "第一步自检发现 X 处逻辑链断裂，已修正如下：[列修改点]。请确认是否同意修改方向，同意后进入写作。"

若第一轮 ✅ 通过或 ❌ ≤ 2 处，直接修正后进入第四步，**不打扰用户**。

> **规则**：此步骤不得跳过（铁律）。MECE 验证通过是进入第四步的必要条件。

#### 修正记录

每次 Step 3.5 发现的框架问题，简要记录在段级论点旁：
- `⚠️ [维度N-问题]: 原论点 → 修正后论点`
- 修正原因：`逻辑链断裂 / MECE 不全 / 重复 / 缺锚点`

此记录最终保留在段簇简报（4a）中，供 4c 拼接时参考。

---

### 第四步：分层撰写

主 agent 内部判断体量（不向用户确认）：

| 条件 | 模式 |
|------|------|
| 成熟蓝筹（数据齐全、无复杂争议）+ 用户未要求特别深度 | **快速通道**：单 `task(make-report, stream_to_parent=true)` 写全部 9 段 |
| 有市场分歧 / 业务复杂 / 可比公司难找 | **编排模式**：4a→4b→4c |

#### 快速通道

加载以下模板文件（按顺序）：`template-datasource.md` → `template-cluster-A.md` → `template-cluster-B.md` → `template-cluster-C.md` → `template-main.md`，将 Step 2 全部数据 + Step 3 段级论点传入 `task(make-report, stream_to_parent=true)` 一次写完 9 段。写完后跳至第五步保存。

#### 编排模式

**4a：主 agent 输出架构终稿**

基于 Step 3 用户确认后的反馈，微调段级论点，形成正式的"段簇简报"。每个简报包含：

```
簇 A 简报：
  - 段1论点：[一句]
  - 段2论点：[一句]
  - 段3论点：[一句]
  - 段4论点：[一句]
  - 数据素材：[iFinD 财务数据 / 行业宏观数据 / 预热结论摘要]

簇 B 简报：
  - 段5论点：[一句]
  - 段6论点：[一句]（含三项子要求）
    * 情绪位置判断：卖方最高/最低 EPS 预测差距、基金重仓占比变化趋势+北向资金持股变化、新闻覆盖频次 vs 历史均值
    * 不对称性赔率：赔率 = (乐观上轨股价 - 当前股价) / (当前股价 - 悲观下轨股价)；>2:1 强不对称性，<1:1 对称赌博或下行更大
    * 具体数据需求：前 10 大流通股东中的基金/北向变化、近 3 个月新闻覆盖总数 vs 上季度、卖方目标价区间
  - 数据素材：[AlphaClaw 研报/纪要摘要 + iFinD 股权/风险数据 + iFinD 行情涨跌幅 + 卖方目标价汇总]

簇 C 简报：
  - 段7论点：[一句]（含安全边际子节要求）
    * 调整后内在价值锚定：存货真实价值（库存吨数×均价）+ 现金及类现金 + 品牌价值 + 永续经营 DCF 估值 → 汇总为调整后每股内在价值
    * 历史安全边际分位：当前调整后 PB（或 PE）处于 2016 年至今的什么分位
    * 安全边际判断：当前价格 vs 调整后内在价值的折价/溢价百分比，结论标注充足/一般/不足
  - 数据素材：[可比公司 iFinD 批量财务数据 + 可比名单 + 存货吨数/均价 + 现金及类现金数据 + 当前股价/市值]
```

**4b：并行段簇写作**

在同一条消息中并行发起 3 个 `task(make-report, mode="subtopic")`：

| Sub-agent | prompt 内容 | 自读文件 |
|-----------|------------|---------|
| Sub-agent A | 簇 A 简报 | `template-datasource.md`（共3步）→ `template-cluster-A.md`（段1-4） |
| Sub-agent B | 簇 B 简报 | `template-datasource.md`（共3步）→ `template-cluster-B.md`（段5-6） |
| Sub-agent C | 簇 C 简报 | `template-datasource.md`（共3步）→ `template-cluster-C.md`（段7） |

Sub-agent 写作规范：
- 不可修改主 agent 给定的段级论点（只能展开）
- 每条结论标注证据等级 + 来源 + 数据时间
- 段内采用金字塔：论点（已给定）→ 子论点 → 数据锚点
- 簇 A 的 sub-agent 按段 1→2→3→4 顺序写，段间加衔接标记
- **禁止生成图片/图表/可视化**：全文仅用 markdown 原生格式（文字、表格、嵌套列表），主体内容 bullet point 驱动

**4c：主 agent 拼接收尾**

读取三个簇的输出文件 → 按 9 段结构拼接，执行以下操作：
1. 写引言（基于首席论点）
2. 补充跨段衔接标记（模板中的"衔接段X"段落）
3. 基于全部段输出，写段 8（跟踪指标）和段 9（待补资料）
4. 统一术语和证据等级标注风格
5. 执行 Acceptance Self-check（内部自检）：对照自检清单逐项验证，发现问题立即修正。**自检过程及表格不输出到最终报告**，仅在全部通过后才交付。若发现无法通过自检修正的问题（如关键数据缺失），回到对应步骤补充
6. 校验一致性：同一数据在不同段中是否一致？段级论点是否都在正文中得到展开？
7. **自动删除中间文件**：确认报告已成功写入后，删除 Step 4b 产生的中间文件（cluster-A.md / cluster-B.md / cluster-C.md）

### 第五步：保存并交付

将报告保存为 Markdown 文件到以下地址：

**保存路径规则**：
```
D:\Alphaclaw\Output\{公司名}-{证券代码}\{yyyymmdd}-{证券代码}-{最新一季财报日期}.md
```

**具体规则**：
1. **子文件夹**：先在 `D:\Alphaclaw\Output\` 下创建以 `公司名-证券代码` 命名的子文件夹（如 `兆易创新-603986`）
2. **文件名**：`{yyyymmdd}-{证券代码}-{最新一季财报日期}.md`，其中：
   - `yyyymmdd` 为报告生成当日的日期（如 `20260508`）
   - `证券代码` 为交易所股票代码（如 `603986`、`688041`）
   - 最新一季财报日期格式为 `YYYYQ{N}`（如 `2026Q1`、`2025Q3`）
3. **覆盖规则**：同一公司、同一财报期若重复研究，**追加内容而非覆盖**，在文件末尾附上新的研究日期和更新内容（保留历史版本）

**示例**：
- 首次研究：文件名 → `D:\Alphaclaw\Output\兆易创新-603986\20260508-603986-2026Q1.md`
- 后续跟踪（同一季度更新）：文件名 → `D:\Alphaclaw\Output\兆易创新-603986\20260510-603986-2026Q1.md`
- 跨季度研究：新文件 → `D:\Alphaclaw\Output\兆易创新-603986\20260820-603986-2026Q2.md`

**交付前清理**：
4. **自动删除中间文件**：最终报告保存成功后，立即删除 3 个段簇中间文件（`cluster-A.md`、`cluster-B.md`、`cluster-C.md`），保持输出目录整洁。若删除失败（如文件仍被占用），记录警告但不阻塞交付

**交付时**：同时告知文件夹路径和文件名，方便后续查找。

## ⚠️ 陷阱速查表

> 详细故障排查 → 读取 `references/troubleshooting.md`

| 严重度 | 陷阱 | 正确做法 | 违反后果 |
|-------|------|---------|---------|
| 🔴 致命 | 管理层公开发言标为 F2 | 业绩会/交流会/管理层公开发言 = M1，不是 F2 | 证据等级污染，结论可信度归零 |
| 🔴 致命 | 可比公司 < 3 家 | 段7 至少 3 家，不足时用 `search_stocks` 补充 | 段7 无横向参照，分析缺乏锚点 |
| 🔴 致命 | 自行估算关键数字 | 数据缺失标注 `[需补]`，段9 列出获取路径 | 幻觉数据侵蚀报告可信度 |
| 🔴 致命 | 编排模式下主 agent 越权写段 1-7 正文 | 主 agent 只定论点 + 拼接收尾，正文由 sub-agent 写 | 并行化失效，单 agent 瓶颈依旧 |
| 🟡 重要 | 段5 纯定性描述（"行业领先""行业龙头"） | 必须引用 Beta/股权集中度/分红等 F2 级数据做量化对比 | 竞争力分析无量化支撑 |
| 🟡 重要 | 段8 写"关注政策变化"等废话 | 必须具体到指标 + 硬触发条件（如"单季净利润 < X亿 → 下调预测"） | 跟踪指标无操作价值 |
| 🟡 重要 | 跳过 Step 3.5 MECE 验证 | 动笔前必须完成 Step 3.5 三维检查，不得以任何理由跳过 | 逻辑链断裂未能在框架层面发现，报告返工成本极高 |
| 🟡 重要 | 行业宏观数据缺失导致段3 空泛 | 用 `get_edb_data` 补充产销量/价格指数数据 | 收入驱动分析缺乏宏观锚点 |
| 🟡 重要 | sub-agent 擅自修改主 agent 给定的段级论点 | sub-agent 只能展开论点，不可推翻或重写 | 首席论点与正文脱节 |
| 🟡 重要 | sub-agent 生成图片/图表/可视化 | 模板文件和 prompt 中显式禁止，Acceptance Self-check 校验 | 报告含非 markdown 原生内容，降低信息密度和可靠性 |
| 🟢 注意 | 研报摘要不足仍写报告 | 用 `get_document_detail` 获取全文 | 段5/段6 分析深度不足 |
| 🟢 注意 | iFinD 工具用裸函数名 | 必须用完全限定名 `服务器名.工具名` | 工具调用失败 |
| 🟢 注意 | 段9 留空 | 基于前 8 段判断，必须列出缺失数据及获取路径 | 数据缺口不可见 |
| 🟢 注意 | 簇间术语不统一（如簇A 写"营收"、簇C 写"收入"） | 主 agent 在 4c 统一术语 | 报告可读性下降 |
| 🟢 注意 | 快速通道场景仍走编排模式 | 成熟蓝筹 + 无争议 → 直接单 agent 写全 9 段 | 增加不必要开销 |
| 🟢 注意 | Acceptance Self-check 表格输出到最终报告 | 自检过程在主 agent 内部完成，修正问题后不保留在交付报告中 | 增加报告冗余内容，降低阅读体验 |
| 🟢 注意 | 中间文件未清理残留 | Step 5 交付后必须删除 cluster-A/B/C.md | 输出目录混乱，后续研究产生同名文件冲突 |
| 🟢 注意 | 段6缺少情绪位置判断 | 必须包含卖方分歧度（EPS预测差距）、持仓拥挤度（基金/北向变化）、媒体热度（新闻频次对比）三个指标 | 市场焦点分析缺乏逆向思维锚点，无法判断当前定价是否已充分反映悲观/乐观 |
| 🟢 注意 | 段6缺少不对称性赔率 | 情景分析末尾必须显式计算赔率 = (乐观上轨 - 当前价) / (当前价 - 悲观下轨) | 情景分析有方向无赔率，无法量化风险收益比 |
| 🟢 注意 | 段7缺少安全边际判断 | 必须包含调整后内在价值锚定（存货+现金+品牌+永续DCF）、历史分位、安全边际百分比 | 可比对标有横向比较无纵向估值判断，报告缺少"当前价格是否值得"的最终锚 |
