# 写入规范

> v1.4 新增。所有知识库条目的格式和质量标准。写入 facts/opinions/conflicts 前必须遵守。

## 事实准入四层过滤

> v1.6 新增。所有事实在写入 facts.md 前必须通过以下 4 层过滤，任何一层未通过不得写入。

```
第1层：信源可靠性
  → 来源是否可追溯？是否为已知可靠信源？
  → 未通过 → 标注"信源未验证"，仅存对话不下发

第2层：交叉验证
  → 同一事实是否被 ≥2 个独立信源确认？
  → 未通过 → 标记为"单一信源待确认"，仍可写入但降级为 F2+

第3层：矛盾分析法评估
  → 该事实是否触及知识库中某个意见(opinion)的主要矛盾？
  → 触及主要矛盾 → 🟡1（写入 facts.md + 标注"待关联"）
  → 未触及任何矛盾 → 🟡2（暂存对话，不写入）

第4层：用户确认
  → 触及主要矛盾的争议事实，必须经用户确认后才写入
  → 纯新事实（🟡2）暂不写入，等被 opinion 引用时升级
```

## 日期标注

所有 facts 和 truths 的 ID 行必须以 `YYYY-MM` 格式标注建立时间：
- facts：`[F-AI-001] 2026-05 [陈述]` — 表示该事实在 2026 年 5 月被首次记录
- opinions：`[T-AI-001] 2026-05 [认知摘要]` — 表示该观点基于 2026 年 5 月的 facts 快照形成

> 日期标记不是"信息发生的日期"而是"信息被沉淀到知识库的日期"。同一 fact 可能在后续月份被新的交叉验证强化并追加验证记录。

## Facts：只追加、不删除

**核心原则**：facts 一经写入，永不直接删除。旧事实不再成立时：
1. 在条目末尾追加"失效标注"（`[已失效 YYYY-MM]：原因`）
2. 在 opinions 层更新修正记录
3. 仅当 **Falsification 检测**通过后，经用户确认，才可删除

**Falsification 检测流程**：
```
发现疑似错误 fact
  → 用 ≥2 个独立数据源交叉验证该 fact
  → 如均确认该 fact 在写入时即为假 → 标记为 [待删除] + 展示验证证据
  → 用户确认 → 删除该 fact + 在修正日志中保留删除记录
  → 如无法确认 → 保留原 fact + 追加"存疑标注"
```

> 注意区分"fact 当时正确但后来被新数据更新"与"fact 写入时即为假"。前者用追加更新 + 旧条目附加失效标注处理；后者才进入 falsification 检测。

## Opinions：多跳链式推导

推导逻辑必须按 `论点1 → 论点2 → … → 结论` 链式格式，每个论点附 L3 证据锚点：

```
推导链路:
  论点1: [陈述] [证据等级]
    L3: [数据锚点]（来源 YYYY-MM）
  论点2: [基于论点1的推进] [证据等级]
    L3: [数据锚点]
  → 结论: [最终判断]
```

> 每个论点回答"为什么这个论点成立"，L3 回答"有什么数据支撑"。整条链从最基础的 facts 逐级推导到最终判断。

---

## 修正记录

修正旧 opinion 时，使用三段式格式：

```
修正记录:
  - YYYY-MM | 修正类型: [补充约束 / 调置信度 / 改写结论 / 新增限制条件]
    原内容: [被修正的原文]
    修正为: [修正后的文字]
    修正理由: [为什么改 + 新增支撑事实 ID]
```

---

## Conflicts：待采集事实 + 裁决闭环

每条 conflict 除了原有字段（对立观点、裁决条件、当前倾向、跟踪记录），**必须增加**：

```
待采集事实:
  - [待采集] [需要采集的事实描述] — 采集后写入 facts.md 并升档为 [F-{topic}-xxx]
  - [待采集] [另一条需要采集的事实]
裁决触发: 当上述 N 条待采集事实中的 M 条被采集并写入 facts.md 后，在步骤 0 中自动触发重新评估
```

> 这使 conflicts 与 facts 形成闭环——conflicts 驱动 facts 的采集优先级，新 facts 的写入触发 conflict 的重新裁决。

---

## Facts 影响索引（可选，渐进实施）

在关键 fact 末尾追加"影响观点"字段，建立从 fact 到 opinion 的反向索引：
```
  影响观点: [T-AI-001]（修正触发） [T-AI-004]（支撑）
```
> 当 fact 发生变化时，可通过此字段快速定位所有受影响的 opinions，触发重新评估。初期仅对关键 facts（如 [F-半导体-012] 这种支撑多条 truths 的节点）标注。

## 新信息引入规则

> 当引入新 facts 或修改旧 facts 时，必须执行以下检查，防止知识库腐烂。

**引入新 fact 时**：
```
1. 交叉验证：≥2 个独立数据源确认 → 确定证据等级
2. 主题归类：属于哪个主题目录？是否需要新建主题？
3. 影响扫描：在现有 opinions 中搜索引用该 fact 提及的实体/指标
   → 如果新 fact 支撑某条 opinion → 追加"影响观点"反向索引
   → 如果新 fact 挑战某条 opinion → 在 opinion 追加变化触发条件 + 必要时写入修正记录
4. 冲突检测：新 fact 是否与已有 fact 矛盾？→ 如是，立即写入 conflicts.md
```

**旧 fact 被更新时**（同一 fact ID，内容扩展或数据刷新）：
```
1. 保留原条目不变，追加更新记录（不删除旧版本）
2. 检查该 fact 的"影响观点"列表：
   → 对每条受影响 opinion：
     - 如果更新支持原推导逻辑 → 无需修改 opinion
     - 如果更新削弱原推导逻辑 → 在 opinion 追加修正记录
     - 如果更新推翻原推导逻辑 → 降低置信度 + 追加修正记录 + 可能触发新的 conflict
3. 更新该 fact 的验证状态和日期
```

**引入新 opinion 时**：
```
1. 多跳链式推导：必须 论点1→论点2→…→结论，每跳有 L3 证据
2. fact ID 引用：每个论点至少引用 1 条已有 fact；如所需 fact 不存在 → 先写入 fact
3. 变化触发条件：明确列出"什么 facts 变化会触发此 opinion 重新评估"
4. 跨主题标注：如引用了其他主题的 fact → 在支撑事实列表后标注（跨主题）
```

> 核心原则：**fact 先于 opinion，opinion 驱动 conflict 的裁决。新信息从 facts 层进入，沿引用链向上传导到 opinions，再横向触发 conflicts 的重新裁决。**
