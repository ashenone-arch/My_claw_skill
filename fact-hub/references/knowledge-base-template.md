## 核心规则（v2.0 更新）

### Facts 只追加、不删除

facts 一经写入，永不直接删除。旧事实不再成立时：
1. 条目末尾追加 `[已失效 YYYY-MM]：[原因]`
2. opinions 层追加修正记录

**Falsification 检测**：仅当 ≥2 个独立数据源交叉验证确认 fact 写入时即为假 → 向用户展示证据 → 用户确认后才可删除。

### 日期标注

所有 facts 和 opinions 的 ID 行必须以 `YYYY-MM` 格式标注建立时间（信息被沉淀到知识库的日期，非信息发生日期）。

## 证据等级体系

### Facts 层

| 标签 | 含义 | 沉淀条件 |
|------|------|---------|
| F1 | 公开可验证事实 | 官方公开渠道，任意人可独立核实 |
| F2 | 金融数据库/审计报告 | iFinD、Wind、年报、公告原文 |
| F2+ | 间接来源与F1交叉验证 | 管理层earnings call、专家访谈中可与F1印证的信息（升档标注） |

### Opinions 层

| 标签 | 含义 | 沉淀条件 |
|------|------|---------|
| O1 | 基于facts的交叉验证判断 | 多个F1/F2相互印证，形成高置信度认知 |
| O2 | 基于facts的合理推断 | 单一方向推演，须附多跳链式推导 |

### Conflicts 层

| 标签 | 含义 | 沉淀条件 |
|------|------|---------|
| C1 | 对立O1/O2判断 | 发现相互矛盾的结论时立即写入 |
| C2 | 待核验假设 | 纯假设性判断，无facts支撑 |

## facts.md：事实层（v2.0）

**格式**：

```markdown
- [F-{大行业}-{细分}-{N}] {YYYY-MM} [陈述]
  来源: [来源名称 + 日期]
  验证状态: 已交叉验证 / 单一信源待确认 / 存疑待核实
  关联实体: [[实体A]] [[实体B]]
  证据等级: F1 / F2 / F2+
  升档依据: [仅F2+填写] 与 [F-xxx-xxx] 交叉验证
  影响观点: [可选，渐进实施] [O-xxx-xxx]（支撑） [O-xxx-xxx]（修正触发）
```

> `影响观点` 字段是 v2.0 的可选字段。标注此 fact 被哪些 opinions 引用、以什么方式引用，便于 fact 变化时快速定位受影响 opinions。

**示例（大行业级）**：

```markdown
- [F-AI-001] 2026-05 Anthropic Q1 2026 ARR 从 $9B 增至 $44B，一个季度翻三倍
  来源: CFO Krishna Rao 公开访谈 2026-05-13
  验证状态: 双源交叉验证（All-In Podcast 2026-05-08 独立确认）
  关联实体: [[Anthropic]] [[Claude Code]]
  证据等级: F2+
  影响观点: [O-AI-001]（支撑） [O-AI-002]（支撑） [O-AI-004]（关联）
```

**示例（细分行业级）**：

```markdown
- [F-AI-基础模型-001] 2026-05 全球基础模型市场 TOP5 厂商合计市场份额约 85%
  来源: CB Insights AI 100 Report 2026-04
  验证状态: 已交叉验证
  关联实体: [[OpenAI]] [[Anthropic]] [[Google DeepMind]]
  证据等级: F1
```

**示例（公司级）**：

```markdown
- [F-AI-基础模型-Anthropic-001] 2026-05 Anthropic 员工总数已超 5,000 人
  来源: LinkedIn 公司主页 2026-05-20
  验证状态: 单一信源待确认
```

## opinions.md：观点层（v2.0 — 多跳链式推导）

**格式**：

```markdown
- [O-{topic}-{N}] {YYYY-MM} [认知摘要]
  类型: O1（交叉验证）/ O2（合理推断）
  置信度: 高 / 中 / 低
  支撑事实: [F-{topic}-{N}] [F-{topic}-{N}]
  推导链路:
    论点1: [陈述] [证据等级]
      L3: [数据锚点]（来源 YYYY-MM）
    论点2: [基于论点1的推进] [证据等级]
      L3: [数据锚点]
    → 结论: [最终判断]
  主要矛盾框架: 属于"XX vs YY"矛盾中的[哪一方]
  变化触发条件:
    - [F-xxx] 若变化 → 需重新评估
    - [具体可观测信号] → 推翻条件
  修正记录:
    - YYYY-MM | 修改类型: [补充约束/调置信度/改写结论]
      原内容: [原文]
      修正为: [新文]
      修正理由: [原因 + 新增支撑事实]
```

**示例**：

```markdown
- [O-AI-001] 2026-05 现阶段 AI 算力供给是收入增长的唯一硬约束
  类型: O1
  置信度: 中高
  支撑事实: [F-AI-001] [F-AI-005] [F-AI-006]
  推导链路:
    论点1: Anthropic ARR 增速远高于算力供给增速 [O1]
      L3: Q1 ARR $9B→$44B 但算力未同比例扩张（CFO Rao 2026-05）
    论点2: 算力供给增加能够即时转化为收入增长 [O2]
      L3: Colossus 1 出租后 Claude 速率限制立即翻倍（All-In 2026-05）
      L3: OpenAI 收入未达标同样源于算力不足（WSJ 2026-05）
    → 结论: 增长的天花板在供给侧而非需求侧
  主要矛盾框架: "供给刚性 vs 需求弹性"，约束在供给侧
  变化触发条件:
    - [F-AI-001] 若连续两季度 ARR 增速放缓而算力供给同步增长 → 需修正
    - [F-AI-005] 若大规模新算力上线后 ARR 增速未同步加速 → 需求瓶颈取代供给瓶颈
  修正记录:
    - 2026-05 | 修改类型: 补充约束
      原内容: 算力供给是收入增长的唯一硬约束
      修正为: 算力供给是硬约束，但资金供给是算力供给的约束——当云厂商 FCF 枯竭时，约束链条可能从需求端断裂
      修正理由: 王川文章 + 三子Agent 交叉验证确认 Amazon FCF 萎缩 95%，新增 [F-半导体-012][F-半导体-013] 支撑
```

## conflicts.md：认知冲突区（v2.0 — 待采集事实闭环）

**格式**：

```markdown
- [C-{topic}-{N}] {YYYY-MM} [冲突描述]
  对立观点:
    - 正方：[判断] — 支撑: [F-xxx] / [O-xxx]
    - 反方：[判断] — 支撑: [F-xxx] / [O-xxx]
  裁决条件: [需要何种证据才能消除冲突]
  待采集事实:
    - [待采集] [需要采集的事实] — 采集后升档为 [F-{topic}-xxx]
 裁决触发: [当 N 条待采集事实中的 M 条被写入 facts.md 后触发重新评估]
  当前倾向: [正方/反方/悬而未决] — [理由]
  跟踪记录:
    - [YYYY-MM] [新事实/新观点]
```

**示例**：

```markdown
- [C-AI-001] 2026-05 Anthropic 与 OpenAI 的"盈亏平衡时间表"冲突
  对立观点:
    - 正方（Anthropic 路径可收敛）：CFO 暗示 2028 年盈亏平衡，基于 ARR 增速 + 推理成本下降 — 支撑: [F-AI-001][F-AI-004]
    - 反方（行业无法收敛）：OpenAI 内部文件显示 2028 年亏损仍达 ~$74B — 支撑: [F-AI-013]
  裁决条件: 需 Anthropic S-1 招股书和 OpenAI S-1 披露的经审计财务数据
  待采集事实:
    - [待采集] Anthropic 经审计的月度经营亏损（S-1 披露后 → [F-AI-xxx]）
    - [待采集] OpenAI 经审计的 2028 年运营亏损预测（S-1 披露后 → [F-AI-xxx]）
  裁决触发: 当上述 2 条待采集事实全部被写入后，自动触发重新评估
  当前倾向: 反方（偏向亏损收敛慢于预期）— CFO 的担忧通常更贴近财务现实
  跟踪记录:
    - 2026-05 王川文章首次提出该矛盾
    - 2026-05-16 综合财务子Agent 确认毛利率从 50% 被下调至 40%
```

## 使用说明

1. **写入规则**：经「来源判断 → 交叉验证 → 升档评估 → 用户确认」四层过滤
2. **facts 更新规则**：只追加、不删除；旧内容标注 `[已失效 YYYY-MM]`；falsification 检测需 ≥2 源确认假 + 用户确认
3. **opinions 更新规则**：新观点追加，旧观点下方添加三段式修正记录
4. **opinions 推导规则**：必须多跳链式（论点1→论点2→…→结论），禁止单跳 [F1]+[F2]→Z
5. **conflicts 闭环规则**：每条 conflict 必须列出待采集事实 + 裁决触发，形成 conflicts→facts 采集的闭环
6. **日期标注**：所有 ID 行必须标注 YYYY-MM 建立时间
7. **引用格式**：使用 `[F-{层级}-{N}]`、`[O-{层级}-{N}]`、`[C-{层级}-{N}]`
8. **跨层引用**：opinions 可引用任意层级的 facts；公司级 opinions 可引用大行业/细分行业的 facts 和 opinions
9. **层级归属**：大行业级 fact 写入大行业 facts.md，细分行业级 fact 写入细分行业 facts.md，公司级 fact 写入公司 knowledge.md
