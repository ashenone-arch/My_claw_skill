# Fact Hub：辩证研究舱 v1.7

## 执行铁律

| # | 规则 | 说明 |
|---|------|------|
| 1 | **知识库优先** | 步骤 0a 先读 index.md + log.md 了解全貌；增量缓存跳过未变更文件；已有数据不重复检索 |
| 2 | **并行** | 步骤 1 定向检索、步骤 3 四路子Agent，必须同一条消息并行发起 |
| 3 | **溯源** | 每条结论标注三要素：证据等级（F1/F2/F2+/T1/T2）+ 来源工具/文件 + 数据时间（YYYY-MM-DD） |
| 4 | **沉淀确认** | opinions 更新必须经用户逐条确认，不可自动覆盖；旧条目标注修正轨迹 |
| 5 | **纯文本** | 全文只用 markdown 原生格式，禁止生成图片/图表/可视化 |
| 6 | **金字塔输出** | 所有步骤输出按 L1（核心判断1句）→ L2（3-5条论据 bullet）→ L3（数据锚点+来源）三层结构 |
| 7 | **分层标注** | 所有输出明确区分：事实 / 真相 / 观点 / 情绪 |
| 8 | **facts 只追加不删除** | facts 一旦写入永不可删除；仅当 ≥2 个独立数据源交叉验证确认 fact 为假时，向用户展示证据，用户确认后才可删除 |
| 9 | **yyyy-mm 日期标注** | 所有 facts 和 opinions 的 ID 行必须以 `YYYY-MM` 格式标注建立时间；facts 可能随新数据出现而变化，opinions 基于某时点的 facts 快照形成 |
| 10 | **facts 写入前交叉验证** | 新 facts 写入前必须通过 iFinD/alphaengine 数据交叉确认（≥2 个独立数据源）。纯定性/方法论框架类 fact 可豁免，但需标注"框架类事实，无需交叉验证"。验证结果标注在 fact 的"验证状态"字段中 |

## 时间精度规则

> 在构造所有数据查询时，必须根据用户的时间表述选择正确的精度。

| 用户表述 | 查询精度 | AlphaClaw date_range | query_finance_data / websearch |
|---------|---------|---------------------|-------------------------------|
| "今天/盘后/盘中/这两天" | 日级别 | `past_day` | 当日日期 |
| "本周/这周/过去一周" | 周级别 | `past_week` | 本周一 ~ 当日 |
| "本月/这个月/近一个月" | 月级别 | `past_month` | 本月1日 ~ 当日 |
| "最近/近期/最新"（无明确时间单位） | 季度级别 | `past_quarter`（默认） | 近3个月 |
| "本季度/近三个月" | 季度级别 | `past_quarter` | 当前季度起始 |
| "近半年" | 半年级别 | `past_half_year` | 近6个月 |
| "近一年/去年" | 年级别 | `past_year` | 对应年份 |

**原则**：用户说了具体时间单位 → 必须用对应精度，禁止扩大范围。步骤 1 定向检索时必须遵守此表。

## 数据源分工原则

| 数据类型 | 首选工具 | 原则 |
|---------|---------|------|
| 结构化数据（财务/估值/行情/风险） | `query_finance_data` | 定量数据不从研报里找数字 |
| 定性分析（研报观点/纪要/行业分析） | `search_finance_reports` | 定性判断不用数据库替代 |
| 实时动态（新闻/政策/热度） | `websearch` | 实时信号不用滞后的研报 |
| 企业工商/股权/风险 | 企业信息类 MCP 工具 | 结构化企业数据不用通用搜索 |
| facts 交叉验证（写入前必做） | iFinD EDB/stock/news MCP（`mcporter call`）+ alphaengine | 参照 equity-deep-research 第二步并行取数模式；与 websearch 形成 ≥2 源交叉 |

> 步骤 1 定向检索时，根据冲突项的数据类型选择对应工具。禁止用错误数据源浪费时间。

## 数据优先级

| 优先级 | 含义 | 说明 |
|-------|------|------|
| P0 | 缺失则判断无法完成 | 冲突项涉及的核心事实数据 |
| P1 | 重要增强 | 冲突项涉及的补充验证数据 |
| P2 | 有则更好 | 背景信息、趋势数据 |

> 步骤 1 定向检索时优先保障 P0 级数据。若 P0 级数据全部缺失，向用户报告后暂停，不自作判断。

## 金字塔分层标准

> 所有步骤输出必须遵循三层 bullet 结构，禁止长篇叙述段落。

| 层级 | 含义 | 粒度 | 格式要求 |
|------|------|------|---------|
| **L1** | 核心判断 | 1-2 句话 | 结论 + 置信度标签（高/中/低） |
| **L2** | 关键论据 | 3-5 条 bullet | 每条 ≤2 行，附证据等级标签（F1/F2/F2+/T1/T2），禁止展开段落 |
| **L3** | 数据锚点 | 每条 L2 下 2-3 条 | 具体数字 + 来源 + 日期，`[F-{topic}-{N}]` 格式引用 |

**示例**：
```
L1: 存储超级周期尚未达盈利峰值，但股价可能已处于顶部区域 — 置信度：中高

L2:
  - 云厂商 Q1 2026 FCF 合计已转负，Capex/经营CF 从 62.9%→86.2% [T1]
    L3: 四家投资CF合计-$246B vs 经营CF $232B（内部数据库 2026-05-16）
    L3: Amazon 净现金仅$240B，不足两季度 Capex（Q1 财报）[F-半导体-005]

  - 2018 年 Micron 股价领先基本面 2 个季度见顶，规律可能重复 [T2]
    L3: Micron 2018年5月峰值$64→年末$28(-57%)（Trefis）[F-半导体-004]
    L3: 当前 SNDK 涨 48x，SOX RSI 达 2000 年互联网泡沫水平（Investing.com 2026-05）

  - 存储涨价已开始挤压客户利润，微软 CFO 公开承认 [F2+]
    L3: $25B Capex 增量直接归因于存储涨价（MSFT FY2026Q3 电话会 2026-04-29）
```

## 概述

Fact Hub 是辩证研究副驾驶，核心功能是**信息的提炼沉淀**与**认知的动态迭代**。

**执行模式 v1.5**：两步摄入（0a 分析对照 → 0b 决策路由）→ 定向检索 → 金字塔预研 → 四路并行 → 极简综合 → 知识库更新（含 log.md 日志 + 步骤 6 自动 lint）。

```
步骤 0：快速对照 + 初始化（知识库优先，冲突评级）
   ↓ 仅 🔴/❓ 项经用户确认后继续
步骤 1：定向信息收集（仅针对冲突项，并行）
步骤 2：金字塔预研框架（L1+L2+L3，含矛盾定位+MECE）
   ↓ 用户确认后继续
步骤 3：四路并行子Agent深度分析（预分配数据包，禁止重复检索）
步骤 4：主Agent综合判定（极简 bullet 正-反-合）
步骤 5：知识库更新（逐项用户确认）
步骤 6：Acceptance Self-check
```

| Conflicts | C1 | 对立T1/T2判断 |

## 写入规范

> 完整规范 → `references/writing-standards.md`。以下为速查摘要：

- **日期**：所有 ID 行标注 `YYYY-MM` 建立时间
- **Facts**：只追加不删除；旧事实标注 `[已失效 YYYY-MM]`；falsification 需 ≥2 源确认假 + 用户确认
- **Opinions**：多跳链式推导 `论点1→论点2→…→结论`；修正用三段式（修正类型/原内容/修正为/修正理由）
- **Conflicts**：必须含 `待采集事实` + `裁决触发`
- **新信息**：fact 先于 opinion；沿引用链传导；引入新 fact 时执行影响扫描→冲突检测

## 交叉验证规范

> v1.7 新增。借鉴 equity-deep-research 第二步"13 条命令并行取数"模式。新 facts 写入前必须经过此流程，确保每条 fact 有 ≥2 个独立数据源支撑。

### 验证流程

```
步骤 0a 提取 🟡1 新事实
  → 步骤 0b 用户确认写入
  → 【交叉验证环节】并行发起 iFinD/alphaengine 查询
  → 验证通过 → 写入 facts.md（标注验证状态）
  → 验证未通过（仅 1 源或无源可验）→ 降级为 F2+ / 标注"单一信源待确认"
  → 验证发现矛盾 → 升级为 🔴 冲突，回到步骤 0b 展示给用户
```

### 工具选择矩阵

| 事实类型 | 首选验证工具 | 备选 | 豁免条件 |
|---------|------------|------|---------|
| 价格/产能/出货量/供需数据 | iFinD EDB `get_edb_data` | websearch | 无公开统计指标 |
| 公司财务/估值数据 | iFinD stock `get_stock_financials` | query_finance_data | 非上市公司 |
| 宏观/行业指标 | iFinD EDB `get_edb_data` | search_finance_reports | 指标不存在于 EDB |
| 企业事件/新闻/公告 | iFinD news `search_news` | websearch | A股以外市场 |
| 产业规律/方法论框架 | 标注"框架类事实" | — | 自动豁免 |

### mcporter call 调用格式

> 引用 equity-deep-research `data-commands.md` 中的格式规范。`tool` 参数格式为 `"服务器.工具"`（点号连接为单个字符串），嵌套在 `args` 内部。

```

  command: "call"
  tool: "hexin-ifind-ds-edb-mcp.get_edb_data"

```

### 验证结果标注

写入 facts.md 时，"验证状态"字段必须标注交叉验证来源：

```
验证状态: 已交叉验证（iFinD EDB 2026-05-16 + websearch 2026-05-16）
验证状态: 框架类事实，无需交叉验证
验证状态: 单一信源待确认（iFinD EDB 未覆盖，仅 websearch 2026-05-16）
```

### 交叉验证豁免

以下类型 fact 可豁免 iFinD 交叉验证，但需在"验证状态"中明确标注豁免原因：
- 产业规律/方法论框架（如"逻辑卖性能、存储卖容量"）
- 学术论文引用（需标注论文全名+发表期刊+年份）
- 非中国的公司/市场数据（iFinD 覆盖范围有限）
- 定性分析框架（如"HBM供给5条路径"）

## 执行步骤

### 步骤 0：研究预热（Broad → Narrow）

**目的**：在对照知识库之前，快速了解外部发生了什么，建立整体事件认知，避免仅凭知识库做判断的盲区。

并行执行以下操作：
1. 调用 `websearch` 查询该主题近期的关键动态（遵守时间精度规则）
2. 调用 `search_finance_reports` 获取相关研报/纪要（如适用）

**预热结论格式**：
```
【研究预热结论】
- 当前核心议题：[1 句话]（来源：[新闻/研报日期]）
- 关键争议点：[存在分歧的核心问题]
- 建议重点验证：[1-2 个需要在知识库对照中重点验证的维度]
```

预热完成后进入步骤 0a。如果预热发现无重大新事件或新闻稀少，继续进入步骤 0a（不阻塞）。

---

### 步骤 0a：增量对照 + 分析

**目的**：先对照已有知识库，判断新材料的增量价值。仅对冲突项启动深度分析，避免全量检索浪费。

**执行流程**：

1. **读取导航入口**：读取 `D:\Alphaclaw\Fact Hub\README.md`（全局索引，含主题概要 + opinion 摘要）+ `D:\Alphaclaw\Fact Hub\log.md`（最近操作记录），快速了解知识库当前状态
2. **初始化检查**：检查 `D:\Alphaclaw\Fact Hub\README.md` 是否存在，不存在则按模板创建知识库结构
3. **增量缓存判断**：检查 log.md 中记录的"上次读取时间"。如果相关主题的 `facts.md` / `opinions.md` / `conflicts.md` 自上次读取后未被修改 → 复用内存中的已有理解，仅读取 index.md 获取更新摘要
4. **读取知识库**（仅首次或文件有变更时）：读取相关主题目录中的 `facts.md`、`opinions.md`、`conflicts.md`
5. **提取断言**：从用户提交的材料中逐条提取可验证的断言
6. **对照匹配**：每条断言与已有知识库条目逐一对比
7. **输出冲突矩阵**（≤10 行 bullet）：

| 标记 | 含义 | 自动处理 |
|------|------|---------|
| ✅ | 与已有 facts 一致 | 跳过，不进入后续步骤 |
| 🟡1 | 新事实，触及次要矛盾 | 追加到 facts.md（标注"待关联"），对话简短记录 |
| 🟡2 | 纯新事实，未触及任何矛盾 | 对话记录暂不写入；当未来被 opinion 引用时升级为 🟡1 |
| 🔴 | 与已有 facts/opinions 冲突，触及主要矛盾 | 进入步骤 0b 展示冲突 |
| ❓ | 无法判断（知识库无相关条目或数据不足） | 进入步骤 0b 展示不确定项 |

**输出格式（金字塔 bullet）**：

```
【快速对照】[文章/材料名称]
L1: 共发现 X 条断言，其中 Y 条与知识库冲突/无法判断

L2（待确认项）:
  - 🔴 [冲突断言] vs 知识库 [T-xxx-xxx]：[冲突说明]
    L3: 文章声称：[具体数据]（来源：文章第N段）
    L3: 知识库记录：[对比数据]（[F-xxx-xxx]）
  - ❓ [无法判断项]：[当前无数据支撑]
    L3: 待采集事实：[需要采集什么数据来消除不确定性]
```

---

### 步骤 0b：决策路由

**目的**：基于步骤 0a 的冲突矩阵，决定是否进入深度分析。

**决策规则**：
- 全 ✅ 或仅 🟡2 → **快速通道**：自动完成（🟡2 暂存对话，记录 log），流程结束
- 仅 🟡1（无 🔴/❓）→ **半快速通道**：追加 facts.md（标注"待关联"）+ 写 log，跳过步骤 1-6，流程结束
- 有 🔴 或 ❓ → **中断**，向用户展示冲突矩阵，等待逐项确认
- 知识库对应主题为空（首次使用）→ 全部标记 ❓，直接询问用户是否全量进入步骤 1

**冲突升级展示（F1 vs T2 冲突）**：

当 🔴 冲突中一方为知识库中已验证的 F1/F2 事实、另一方为新材料中的 T2 推断时，在冲突矩阵中**特别标注**：

```
  - 🔴 [F vs T] [冲突断言] vs 知识库 [T-xxx-xxx]
    L3: ⚠️ 知识库已验证事实：[具体内容]（[F-xxx-xxx]，已验证）
    L3: 新材料推断：[具体内容]（来源：文章第N段，未验证）
    L3: → 请确认以哪方为准。知识库事实经过了交叉验证流程，但新材料可能包含我们未覆盖的信息
```

> **核心原则**：事实与推断的冲突**不自动裁决**，一律展示给用户确认。用户掌握最终判断权。

**用户确认 🔴/❓ 项后**，进入步骤 1。

### 步骤 1：定向信息收集

> 执行前读取 `references/data-collection-guide.md`。

**仅针对步骤 0 确认的 🔴/❓ 项进行检索**，不对全文做全量搜索。

在同一条消息中并行发起检索：

- **结构化数据**：`query_finance_data` → 仅查冲突/不确定项涉及的关键指标
- **定性分析**：`search_finance_reports` → 仅搜索冲突项相关主题
- **实时动态**：`websearch` → 仅补充时效性数据

> 铁律：所有检索必须在同一条消息中并行发出；禁止检索步骤 0 中已标记 ✅ 的内容。

### 步骤 2：金字塔预研框架

> 执行前读取 `references/pre-research-template.md`。

基于步骤 0-1 的结果，输出金字塔结构预研框架（直接输出，不生成单独文件）：

**必须包含以下板块（全部按 L1→L2→L3 三层 bullet）**：

1. **矛盾定位**（L1：1句矛盾判断 → L2：矛盾对列表 → L3：数据支撑）
2. **重要性判决**（L1：行动级/观察级 → L2：判决依据 3 条 → L3：证据锚点）
3. **MECE 自检**（4 项 checklist，每项 1 行）
4. **子Agent 分配方案**：按 🔴/❓ 项涉及的维度，选择 1-4 个子Agent（地缘/产业/技术/财务），每个子Agent 赋予专属的"已知数据包"（步骤 0-1 已获取的数据 + 知识库相关条目 ID）
5. **反转追问**（L1：共识是什么 → L2：共识可能错在哪个维度 → L3：什么证据出现会推翻判断）

#### 塔尖断言（必做）

在子Agent 分配方案中，为每个选定的子Agent 维度预写一句**塔尖断言**：

```
L2（塔尖断言）:
  - 地缘维度塔尖：[1 句断言]
  - 产业维度塔尖：[1 句断言]
  - 技术维度塔尖：[1 句断言]
  - 财务维度塔尖：[1 句断言]
```

**写作规范**：
- 每句用断言句，不用主题描述句。❌ "地缘政治维度分析" ✅ "台海安全焦虑已将半导体供应链从效率优先转向安全优先"
- 各维度塔尖之间逻辑自洽，不相互矛盾
- 如果某维度数据不足无法形成断言，标注 `[待数据补充]`，不编造

> 这些塔尖断言将作为步骤 3 子Agent 写作时的"维度塔尖"——子Agent 只能展开论证，不能推翻。

**向用户确认后**，进入步骤 3。

---

### 写作前综合检查（确认后、步骤 3 前必做）

> 确认后、启动子Agent 前，先自问以下 5 条。任一条为"否"则先补充检索，不急于动笔。

| # | 检查项 | 通过标准 |
|---|-------|---------|
| 1 | 多角度覆盖 | 已从 ≥3 个角度检索（事实/研报/新闻，对应步骤 0+1）？ |
| 2 | 核心数据就位 | 🔴 冲突项涉及的核心事实数据覆盖率 ≥ 80%？每个关键冲突有 F1/F2 级数据锚点？ |
| 3 | 正反观点完整 | 同时检索了支持知识库观点的证据和挑战知识库观点的证据？ |
| 4 | 核心冲突可回答 | 步骤 0 发现的核心 🔴 冲突有数据可以推进裁决？ |
| 5 | 信息时效合格 | 检索数据来自正确的时间范围（遵守时间精度规则）？知识库引用的事实均在有效期内（未标记 [已失效]）？ |

→ **任一条未通过**：先补充检索，不急于启动子Agent。全部通过后进入步骤 3。

---

### 步骤 3：四路并行子Agent 深度分析

> 执行前读取 `references/subagent-prompts.md`。

在同一条消息中并行发起 1-4 个子Agent（根据步骤 2 的分配方案选择）：

| Sub-agent | 负责维度 | 分析框架 | 禁止触碰 |
|-----------|---------|---------|---------|
| A：地缘 | 安全-资源-阵营三角 | 行为体安全焦虑 / 资源通道定价 / 阵营重组 | 产业供需 / 技术路线 / 财务数据 |
| B：产业 | 供需-格局-周期三轴 | 供需缺口方向 / CR3/CR5 趋势 / 周期阶段定位 | 地缘政治 / 技术路线 / 财务分析 |
| C：技术 | Science-Engineering-Economics | 阶段判断 / 降本路径 / 替代威胁 | 地缘政治 / 产业格局 / 利润质量 |
| D：财务 | 经营实质还原 | 利润质量+激励对齐+资产质量 / FCF 可持续性 / 估值锚点 | 地缘政治 / 技术路线 / 产业周期 |

**每个子Agent 的 prompt 必须包含**（由主Agent 步骤 2 预组装）：

1. 研究主题 + 首席判断（L1）+ 矛盾定位
2. **已知数据包**：该维度对应的已检索数据 + 知识库引用 ID（禁止子Agent 重复检索）
3. 专属分析维度（仅限该子Agent 负责的范围）
4. 输出格式约束：L1（核心判断 1 句 + 对首席判断的支撑/修正/推翻）→ L2（3-5 条论据 bullet，附证据等级）→ L3（每条论据下 2-3 条数据锚点 + 知识库引用）
5. 每个子Agent 必须明确回答：对步骤 2 首席判断是**支撑、部分修正还是推翻**？
6. 每个子Agent 的维度塔尖断言由主Agent 在步骤 2 给定，子Agent 只能展开论证，不可推翻

**反向探测（输出末尾必做）**：

每个子Agent 的输出末尾必须包含：
```
L2（反向探测 — 我们遗漏了什么）:
  - 在本维度分析中，还有什么重要问题我们没有覆盖但你认为很重要？
  - 如果有新证据在 [具体条件] 下出现，会如何修正本维度的判断？
```

**铁律**：
- 四路子Agent 必须在同一条消息中并行发起
- 禁止交叉检索：A 不能碰 B/C/D 的维度，B 不能碰 A/C/D 的维度
- 必须引用知识库 ID（`[F-xxx-xxx]` / `[T-xxx-xxx]`），知识库为空时标注"知识库为空，以下为首次建立"

---

### 步骤 4：主Agent 综合判定

读取 1-4 个子Agent 的输出，输出极简综合判定（全部 bullet，禁止段落）。核心原则：**只输出步骤 2 中不存在的增量认知，不重复子Agent 已有的论据**。

**输出格式**：

```
【综合判定】[主题]

L1: [1 句话核心结论]（置信度：高/中/低）

L2（正-反-合，各 3-5 bullet）:
  正方（支撑增长叙事的论据）:
    - [论据 1] [T1/F2]
    - [论据 2] [F1]
    ...

  反方（挑战增长叙事的论据）:
    - [论据 1] [T1/F2+]
    - [论据 2] [F2]
    ...

  合方（认知跃迁，在更高层面统一矛盾）:
    - [跃迁认知] [T2]
    ...

L2（关键观测指标，3-5 个）:
  - 指标：[触发条件] → [预期结果]
  - 指标：[触发条件] → [预期结果]

L2（认知更新提案）:
  - 新增真相 [T-{topic}-{N}]：[摘要]（置信度）
  - 修正真相 [T-{topic}-{N}] → [修正后]
  - 新增冲突 [C-{topic}-{N}]：[摘要]
    L3: 待采集事实：[裁决此冲突需要采集的事实]
    L3: 裁决触发：[当哪些待采集事实被写入后触发重新评估]
```

---

### 步骤 5：知识库更新

向用户逐条展示步骤 4 的认知更新提案，逐项确认：

```
本次知识库更新提案：

■ 新增事实 [x条]：[F-{topic}-{N}] [陈述] → 交叉验证状态：[已交叉验证/框架豁免/单一信源] → 写入 {topic}/facts.md？

■ 新增真相 [x条]：[T-{topic}-{N}] [认知摘要] → 写入 {topic}/opinions.md？

■ 修正真相 [x条]：[T-{topic}-{N}] → [修正后] → 旧版本保留修正轨迹

■ 新增冲突 [x条]：[C-{topic}-{N}] [冲突描述] → 写入 {topic}/conflicts.md？
     待采集事实：[列出裁决此冲突需要的事实] 裁决触发：[条件]

■ 失效事实 [x条]：[F-{topic}-{N}] → 标注 [已失效 YYYY-MM] + 原因 → 追加而非删除

■ 待删除事实 [x条]：[F-{topic}-{N}] → ≥2 源确认假 → 请确认删除？[Y/N]
```

用户逐项确认后，执行文件写入。

**写入后追加日志**：每次知识库更新完成后，向 `D:\Alphaclaw\Fact Hub\log.md` 追加一条操作记录：
```
- YYYY-MM-DD HH:MM | 操作类型: [ingest/update/correct/delete] | 主题: [AI/半导体] | 涉及 ID: [F-xxx-xxx] 等 | 摘要: [1 句话说明]
```

### 步骤 6：自动 Lint + Self-check + 中间文件清理

> 执行前读取 `references/self-check.md`。

**不再手工 checklist**。解析所有知识库文件中的 fact ID 引用，自动执行以下检查：

```
【自动 Lint】

■ 破引用检测：
  ✅/⚠️ 所有 opinion/conflict 引用的 [F-xxx-xxx] 在 facts.md 中存在？
  ✅/⚠️ 所有 conflict 引用的 [T-xxx-xxx] 在 opinions.md 中存在？
  ⚠️ 发现 X 条未定义引用：[列出]

■ 孤立条目检测：
  ✅/⚠️ 每条 fact 是否被至少 1 条 opinion 引用？（孤立 fact = 信息未被利用）
  ⚠️ 孤立 facts（无 opinion 引用）：[列出]

■ 悬空引用检测：
  ✅/⚠️ 每条 opinion 引用的 fact 是否未标记 [已失效]？
  ⚠️ opinion 引用了已失效 fact：[列出]

■ 过期修正检测：
  ✅/⚠️ 修正记录中的 facts 是否仍处于旧状态？
  ⚠️ 修正记录对应的 fact 已更新但 opinion 未同步：[列出]

■ 稀疏主题检测：
  ✅/⚠️ 同一主题内 facts 之间是否有交叉引用？（稀疏 = 信息孤岛）
  ⚠️ 稀疏主题：[列出]
```

输出单段结论：

```
【Self-check】
✅/⚠️ 破引用(X/Y) | ✅/⚠️ 孤立条目(X) | ✅/⚠️ 悬空引用(X) | ✅/⚠️ 过期修正(X) | ✅/⚠️ 稀疏主题(X)
[如有 ⚠️，逐条 1 行说明修正方案]

■ 中间文件清理：
  ✅ 已清理 / ⚠️ 清理失败
```

**中间文件清理**：
- 步骤 6 完成后，检查步骤 3 子Agent 是否产生了中间文件（如 `.alphaclaw/tmp/` 下的临时分析文件）
- 如有，确认知识库更新已保存后，自动删除这些中间文件
- 若删除失败（如文件仍被占用），记录警告但不阻塞交付

## 流程路由

| 用户输入 | 路由 | 执行步骤 |
|---------|------|---------|
| 提交研究材料/报告 | **流程 A：信息沉淀** | 0a → 0b → (如有🔴/❓)1 → 2 → 3 → 4 → 5 → 6 |
| 输入突发新闻/事件 | **流程 B：事件驱动** | 0a（重点在矛盾定位）→ 0b → (如有🔴)1 → 2 → 3 → 4 → 5 → 6 |
| 开放式研究问题 | **流程 C：问题求解** | 0a → 0b → 1 → 2 → 3（四路并行）→ 4 → 5 → 6 |
| 步骤 0a 仅 ✅/🟡 | **快速通道** | 0a → 0b → 自动完成（追加 facts + 写 log，不进入步骤 1-6） |
| 步骤 0a 仅 ✅/🟡/🟡1 | **半快速通道** | 0a → 0b → **facts 交叉验证（iFinD/alphaengine）** → 追加 facts + 写 log，跳过步骤 1-6 |
| 仅 1-2 项 🔴/❓ + 简单数据需求 | **轻量通道** | 0a → 0b → 1（≤3 条并行检索）→ 2（轻量预研）→ 跳过 3-4 直接综合判定 → 5 → 6 |

## 陷阱速查表

| 严重度 | 陷阱 | 正确做法 |
|-------|------|---------|
| 🔴 致命 | S3/T2 以 S1/F1 口吻沉淀 | T2 必须附推演链路，禁用断言式语气 |
| 🔴 致命 | 旧 opinion 被覆盖而非标注修正 | 新 opinion 仅追加，旧条目追加"修正记录" |
| 🔴 致命 | 真相更新未经用户确认 | 步骤 5 必须逐项用户确认 |
| 🔴 致命 | 跳过步骤 0 直接全量检索 | 必须先对照知识库，冲突确认后再检索 |
| 🟡 重要 | 子Agent 重复检索已有数据 | 主Agent 预分配"已知数据包"，子Agent 禁用检索 |
| 🟡 重要 | 子Agent 越界分析 | 四维度严格隔离，A↛B/C/D |
| 🟡 重要 | 输出非金字塔结构 | 所有步骤严格按 L1→L2→L3 三层 bullet |
| 🟡 重要 | 重要性判决跳过矛盾分析 | 必须先矛盾定位，再重要性判决 |
| 🟡 重要 | 跳过写作前综合检查 | 步骤 3 前必须执行 5 条检查，任一条未通过 → 先补充检索 |
| 🟡 重要 | 子Agent 擅自修改塔尖断言 | 子Agent 只能展开论证，不可推翻或重写主Agent 给定的维度塔尖 |
| 🟡 重要 | 检索不遵守时间精度规则 | 步骤 1 必须根据用户时间表述选择对应精度，禁止为"多召回"而扩大范围 |
| 🟡 重要 | F1 vs T2 冲突被自动裁决 | 必须展示给用户确认，不可由系统自行判定 |
| 🟡 重要 | 跳过研究预热直接对照知识库 | 步骤 0 必须先了解外部发生了什么，再对照知识库 |
| 🟢 注意 | 认知冲突未写入 conflicts.md | 发现对立证据立即写入，不等待裁决 |
| 🟢 注意 | 子Agent 未在同一条消息并行 | 四路必须一次发出，不得串行 |
| 🟢 注意 | 步骤 4 重复步骤 2 已有论据 | 综合判定仅输出增量认知跃迁 |
| 🟢 注意 | 中间文件未清理残留 | 步骤 6 完成后应删除步骤 3 产生的临时文件 |
| 🟢 注意 | 子Agent 未执行反向探测 | 每个子Agent 输出末尾必须包含"我们遗漏了什么"自查 |
| 🔴 致命 | facts 被直接删除而非标注失效 | facts 只追加；旧内容失效时标注 [已失效 YYYY-MM]；仅 falsification 检测通过 + 用户确认后才可删除 |
| 🔴 致命 | opinions 推导逻辑是单跳而非多跳链式 | 必须 论点1→论点2→…→结论，每跳有 L3 证据；禁止 [F1]+[F2]→Z 直接跳跃 |
| 🟡 重要 | conflict 无待采集事实和裁决触发 | 每条 conflict 必须列出待采集事实 + 明确裁决触发条件 |
| 🟢 注意 | facts/opinions 缺少 yyyy-mm 日期 | 每条 ID 行必须标注建立时间的 YYYY-MM |
| 🟡 重要 | 破引用：opinion 引用了不存在的 fact | 步骤 6 lint 自动检测，修正后重新验证 |
| 🟢 注意 | 孤立 fact：未被任何 opinion 引用 | 步骤 6 lint 报告孤立条目，评估是否需要新增 opinion 或标注该 fact 为待利用 |
| 🟢 注意 | 稀疏主题：同主题内 facts 无交叉引用 | 步骤 6 lint 提示，考虑是否需要建立 fact 间关联 |
| 🔴 致命 | facts 未经 iFinD/alphaengine 交叉验证直接写入 | 非豁免类新 fact 写入前必须通过 iFinD/alphaengine 交叉验证（≥2 源）；验证流程见"交叉验证规范" |
| 🟡 重要 | 交叉验证发现矛盾但被静默忽略 | 验证发现与知识库矛盾 → 必须升级为 🔴 冲突，展示给用户确认，不可自行裁决 |
