# 交叉分析框架工具箱

> 按需加载。当步骤 5 遇到复杂交叉分析场景（多方观点难以形成清晰判断）时读取本文件。
> 不需要全部读完，按场景索引查找。

---

## 框架选择速查

| 场景 | 推荐框架 | 解决什么问题 |
|------|---------|-------------|
| 多方观点看似矛盾 | 立场光谱 | 理解分歧根源（信息差/立场差/时间差） |
| 多方观点高度一致 | 共识边界 | 识别共识的具体范围和盲区 |
| 同一话题在不同时间有演化 | 时间演变轴 | 追踪观点随事件/时间的漂移 |
| 需要从多篇中提炼深层主题 | 元主题提取 | 识别嘉宾未明说但共同指向的命题 |
| 发现某话题只有一方谈及 | 结构洞识别 | 标记信息缺口，避免以偏概全 |

---

## 框架详解

### 1. 立场光谱

**用途**：当多篇对同一话题给出不同甚至矛盾的观点时，将各方立场沿一个或两个维度排列，识别分歧的根因。

**核心问题**：分歧是因为信息不同？立场/利益不同？还是时间/市场条件不同？

**操作步骤**：
1. 提取各嘉宾在该话题上的核心主张（Q1/Q2）
2. 判断分歧类型：
   - **信息差**：双方所知的事实不同 → 分歧可通过信息互通消除
   - **立场差**：双方利益/角色不同 → 分歧是结构性的，不太可能调和
   - **时间差**：访谈时间不同，市场/技术条件已变化 → 后发观点可能已过时
3. 画出立场轴：
```
立场轴：悲观 ─────────────────────────────── 乐观
嘉宾A  ████████████
嘉宾B                    ████████████
嘉宾C                          ████████████
```

**输出格式**：
```
### 立场光谱分析
- 分歧类型：[信息差/立场差/时间差]
- 根因：[一句话]
- 推论：若[条件变化]，[哪方立场]将向[方向]移动（Q3）
```

### 2. 共识边界

**用途**：当多方观点高度一致时，不满足于"达成了共识"，而是精确画出共识的边界——大家到底在什么具体点上一致、什么点上可能已有分歧但未展开。

**核心问题**：共识的边界在哪里？这个边界之外有什么？

**操作步骤**：
1. 列出各嘉宾明确一致的点（Q1/Q2）
2. 列出各嘉宾**未表态**但在其他文件中被讨论的点
3. 判断：未表态是因为"不反对"还是"没被问到"？

**输出格式**：
```
### 共识边界分析
- 共识核心：[具体一致的点]（覆盖 X/Y/Z 文件）
- 共识边界：[此处是未讨论/未明确的部分]
- 边界外的风险：若[边界外条件变化]，共识可能破裂（Q3）
```

---

### 3. 时间演变轴

**用途**：当同一话题在多篇不同时间的文件中被讨论时，追踪观点如何随事件/市场条件演变。

**核心问题**：观点变化是因为新信息出现？还是嘉宾本身立场在漂移？

**操作步骤**：
1. 按时间排列各文件中该话题的表述
2. 标注每个时间节点的关键外部事件（政策发布、财报、产品发布等）
3. 判断：观点变化与外部事件的因果关系

**输出格式**：
```
### 时间演变轴

| 时间 | 文件 | 嘉宾 | 核心表述（Q1/Q2） | 同期关键事件 |
|------|------|------|-------------------|-------------|
| 2025-03 | file1 | A | "..." | 事件X |
| 2025-09 | file2 | B | "..." | 事件Y |
| 2026-01 | file3 | A | "..." | 事件Z |

- 演变方向：[观点从X→Y]
- 驱动力：[事件A是关键转折点]（Q3）
```

### 4. 元主题提取

**用途**：当多篇文件各自讨论不同子话题，但深层共同指向某个更高层次的主题时使用。这个元主题通常嘉宾本人没有明说。

**核心问题**：多篇文件共同指向了什么更大的命题？

**操作步骤**：
1. 列出各文件的表层话题标签
2. 寻找交叉点的模式：它们都在回答什么问题？
3. 提炼元主题（注意：元主题必须是 Q3，不能包装为嘉宾的观点）

**输出格式**：
```
### 元主题
- 表象话题：A 谈芯片性能，B 谈软件生态，C 谈客户获取
- 共同指向：三者都在回应同一个问题——[XX行业]的瓶颈已从[硬件]转移到[生态和商业化能力]
- 推理链路：A 说性能已过剩（Q1-[fileA]），B 说客户最关心易用性而非算力（Q1-[fileB]），C 说签约周期变短说明决策链在简化（Q2-[fileC]）→ 三者共同指向硬件不再是瓶颈（Q3）
```

---

### 5. 结构洞识别

**用途**：识别某个重要话题只有部分嘉宾谈及、或者关键角色缺席（如只有供应商视角没有客户视角）。

**核心问题**：谁的观点缺失了？这个缺失对整体判断有什么影响？

**操作步骤**：
1. 对每个话题，列出已覆盖的视角类型（供应方/需求方/竞争者/监管等）
2. 标注缺失的视角
3. 判断缺失的严重度：是否影响核心判断？

**输出格式**：
```
### 结构洞
- 已覆盖视角：[嘉宾类型A、B]
- 缺失视角：[嘉宾类型C]
- 影响：[若 C 的观点与 A 相反，则[A的核心判断]需要重新评估——但目前无法验证]（Q3）
```

## 使用原则

1. **不必每个话题都用框架**——简单场景（"A 说 X，B 也说 X，两人例子不同但逻辑一样"）直接金字塔即可
2. **一个话题最多用 2 个框架**——框架是辅助判断的工具，不是必须展示的格式
3. **框架分析嵌入"交叉分析"小节**——不作为独立章节，而是金字塔底层的支撑逻辑
4. **所有框架输出都是 Q3**——必须标注推理前提和链路
