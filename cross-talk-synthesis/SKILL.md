# 多篇对谈交叉汇总

## 概述

将用户提供的多篇已有对谈文章（通常由字幕转文章流程生成），按**共同话题**重新组织为一篇大型交叉分析文章。输入是多篇已完成的 Markdown 文章文件，输出是以话题为轴心组织、汇集各方观点的总结文章。

与 youtube-transcript-to-article 的区别：本 Skill 不涉及字幕下载、VTT 解析、分块处理等工程步骤。纯「读取已有文章 → 识别交叉话题 → 按话题重组写作」。

## 使用场景

- 多期播客/访谈围绕同一大主题（如 AI、地缘政治、某种技术趋势），用户希望看到不同嘉宾在关键子话题上的观点碰撞
- 同一话题跨越数月的讨论，用户希望做时间维度的交叉比较
- 多篇深度采访需要提炼出共性的洞察和分歧点
- 用户说"这几篇都在聊 X，帮我按话题整理一份总结"
- 用户提供一个目录，希望 AI 读取其中全部报告并生成汇总文章（无论文件格式是 PDF 还是 Markdown）

## 执行步骤

### 步骤 1：确定输入文件

通过以下方式之一获取文章文件列表：
- 用户直接列出文件路径
- 用户指定某个目录，用搜索工具获取该目录下所有文件（支持 .md、.pdf 等格式）
- 用户用自然语言描述（"上次生成的那几篇 Jeff Kaplan 和 Dylan Patel 的文章"），搜索工作区定位

**识别文件类型**：
- `.md` → 直接读取
- `.pdf` → 使用 `pdfplumber` 库提取文本（见"附录：PDF 读取说明"）
- 混合类型 → 分别处理后汇总

向用户展示文件列表确认。如果文件数量过多（>10 篇），建议用户筛选聚焦。

### 步骤 2：结构化提取（并行）

**不要将多个文件合并为一个纯文本文件**，而是**每个文件独立输出结构化摘要**，便于后续精准定位。

对每个文件并行执行以下提取，使用 task 工具启动 subagent：

对每个文件，提取以下结构化信息：
- `title`：文件名（去除日期和后缀后的语义名称）
- `date`：文件日期
- `expert_background`：受访者背景（职位、专注领域、可讨论话题）
- `key_topics`：该访谈覆盖的主要话题列表（用自然语言描述，不要编号）
- `key_points`：每个话题下的核心观点和关键数据（每条附上原文位置引用，便于后续回查）

**格式要求**：输出 JSON 结构，每篇文件生成一个独立的摘要对象。最终汇总到一个 JSON 数组，保存到 `.alphaclaw/tmp/{主题}_结构化摘要.json`。

**并行原则**：文件数量 ≥ 3 时，必须在**同一条消息中并行**发起所有 subagent，不逐个串行。

示例结构：
```json
[
  {
    "title": "QQ音乐品牌总监视角下的版权格局",
    "date": "2025-08-07",
    "expert_background": "腾讯音乐娱乐集团前任QQ音乐品牌总监，聚焦品牌战略和用户画像",
    "key_topics": ["版权壁垒逻辑", "汽水音乐用户侵蚀", "QQ音乐商业化策略", "粉丝经济"],
    "key_points": {
      "版权壁垒逻辑": {
        "核心观点": "版权是壁垒，但壁垒的意义在于粘性而非获客。",
        "关键数据": "TME曲库规模2.8亿，汽水音乐5000万。",
        "原文位置": "第4页"
      }
    }
  }
]
```

### 步骤 3：识别并确认交叉话题（必须执行）

基于所有文件的结构化摘要，统一识别**交叉话题**——即在两篇（含）以上文件中被实质讨论的主题。

**话题识别原则**：
- 话题应该是**子话题级别**（如"模型训练成本""开源与闭源之争"），而非笼统大主题（如"AI"）
- 一篇长文章中通常有 5-15 个可识别的子话题
- 识别条件：至少在两篇中被实质讨论（不是一笔带过）才算交叉话题
- 排除：纯个人叙事（某一嘉宾的个人故事，其他嘉宾没有对应讨论）不作为交叉话题，但可在引言或结语中作为单篇亮点提及

**生成话题清单**，格式如下，**必须展示给用户确认**（不允许跳过此步骤）：

```
## 识别到的交叉话题

1. 【版权壁垒逻辑】
   涉及：QQ音乐品牌总监（TME）、TME商业化负责人、网易云曲库负责人
   核心分歧：版权是获客壁垒还是粘性壁垒？

2. 【汽水音乐增长策略】
   涉及：QQ音乐品牌总监（TME）×2、TME商业化负责人、网易云曲库负责人
   核心分歧：汽水音乐能否独立商业化？

3. ...
```

**允许用户增删或调整话题优先级**，根据用户反馈确定最终撰写范围。

### 步骤 4：按话题分组并行提取相关段落

对每个确认的交叉话题，启动 subagent 从对应文件中提取该话题下的所有相关段落。

**并行原则**：所有话题在**同一条消息中并行**提取，不逐个串行。

每个话题的提取 prompt 中需包含：
- 话题名称和具体提问
- 涉及哪些文件（文件路径和结构化摘要路径）
- 要求提取：直接引用原文 + 嘉宾姓名 + 关键数据或事实
- 要求标注"一致性"或"分歧"——如果多位嘉宾观点一致，也如实说明

### 步骤 5：撰写总结文章

基于步骤 4 的结构化提取结果撰写总结文章。

每个话题段落须包含：
- 该话题下各嘉宾的核心观点（标注来源嘉宾和文件）
- 观点的一致性和分歧（分歧本身是信息，不要为了叙事流畅而抹平）
- 每个话题的长度与原文讨论深度正相关

### 步骤 6：保存文章

将文章写入用户指定路径。若用户未指定，默认保存至工作目录，文件名格式：`{主题}-多篇对谈交叉汇总.md`。

## 文章撰写标准

以下标准继承自 youtube-transcript-to-article 的改进版本，适用于本 Skill：

**保真底线**：
- 不凭空编造任何事实、数据或观点；不歪曲任何嘉宾的立场和原意
- 允许在同一段内对纯粹的口语重复进行压缩合并，但不削去携带新信息维度的表述
- 跨文件引用同一话题的表述时，忠实保留每位嘉宾的原话质地，不因归纳而丢失差异性

**信息增量测试**（判断一段话是否可以压缩）：
- 携带新事实数据 → 保留
- 添加新分析维度 → 保留
- 增加限定条件 → 保留
- 给出因果解释 → 保留
- 有具象化场景/例子 → 保留
- 纯粹换措辞复述、口语填充、自我修正中间表述 → 可压缩
- 兜底原则：拿不准时默认保留

**主持人话语处理**：
- 原文中主持人功能性互动（附和、简短过渡）去除
- 内容性提问/评论转为叙事引导句，避免 Q&A 格式

**叙事性 vs 说明性双轨**：
- 叙事性内容（嘉宾亲述的个人经历、关键转折时刻、情绪高峰）→ 使用直接引语，保留冲击力
- 说明性内容（概念解释、背景、系统原理）→ 精炼书面语准确归纳，受信息增量测试约束

**极简过渡句**：
- 话题之间可加 1-2 句过渡，仅用于桥接上下文的跳跃，不凭空编造话题间关系

**专有名词标注**：
- 人名、公司名、产品名、技术术语、特定概念在首次出现时用中文后括号注明英文原文
- 后续同一名词可只写中文

**多篇特有规则**：
- 每个话题段落开头必须注明嘉宾姓名，让读者知道谁的发言
- 观点分歧必须呈现，不为了叙事流畅而让"各方似乎达成共识"——如果原文中观点确实一致，也如实说明"A 和 B 在这一问题上观点一致"
- 不添加原文中没有的"分析""评价""总结"——文章的价值在于把多篇原文中的共通脉络展示出来，而非代替读者做判断

**信息来源标注规范**：
- 每个子话题开头用"**来源覆盖**：[嘉宾A-文件1]、[嘉宾B-文件2]"标注信息来源
- 关键数据后用括号标注来源和日期，如"（TME商业化负责人，2026年2月）"
- 避免在文中使用"据采访显示"等模糊表述，应具体到哪位嘉宾、哪份文件

## 文章结构模板

```markdown
# {大主题}——多篇对谈交叉分析

## 引言
简要说明涉及的嘉宾、各自背景、对话发生的上下文。1-3 段。

## 一、{子话题 1}
**来源覆盖**：[嘉宾A-文件X]、[嘉宾B-文件Y]

### A 的观点
（附关键引语和数据）

### B 的观点
（附关键引语和数据）

### 交叉分析
（一致性和分歧）

## 二、{子话题 2}
（同上）

...

## 结语
可以简要提及各篇中未能形成交叉的个人亮点故事（如果有值得保留的），以示对每篇原文的尊重。
```

模板为建议结构，可根据实际话题数量灵活调整。

## 注意事项

- 本 Skill 用于「多篇→一篇」的汇总。如果用户只有一篇文章要处理，引导其使用其他适合的工具
- 如果文章数量极大（>10 篇），主动建议用户缩小范围或分批处理，避免单一文章信息密度过低
- 交叉话题数量建议控制在 5-10 个——太少说明话题粒度过粗，太多说明每篇信息量不足
- 不要因追求「交叉」而强迫关联——如果某篇文章讨论的议题在其他文章中完全没有对应，不要生硬关联，可在引言或结语中单篇提及

## 附录：PDF 读取说明

当输入文件为 PDF 时，按以下步骤处理：

**第一步：检查 Python 环境中的 PDF 库**

在命令行执行：
```bash
python -c "import pdfplumber; print('ok')"
```

如果报错 `ModuleNotFoundError`，先安装：
```bash
python -m pip install pdfplumber --quiet
```

**第二步：读取 PDF 内容**

使用 pdfplumber，避免使用 pymupdf（在该环境下静默失败，难以排查）。读取示例：
```python
import pdfplumber, os

files = [列表所有PDF路径]
with pdfplumber.open(file_path) as pdf:
    for i, page in enumerate(pdf.pages):
        text = page.extract_text()
        if text and text.strip():
            # 追加到输出
```

**中文路径处理**：Python 3.13+ 原生支持中文路径，直接传入绝对路径字符串即可，无需额外编码处理。

**第三步：提取后立即结构化**

不要将 PDF 文本直接追加到纯文本文件——每读完一个 PDF，立即提取结构化摘要并追加到 JSON 文件中，避免大量文本堆积在内存中导致后续处理困难。

**第四步：文件过大的分页处理**

如果单个 PDF 超过 20 页，按每 10 页为单位分段读取，避免单次提取超时。
